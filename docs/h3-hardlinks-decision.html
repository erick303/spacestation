H3 — DirSize double-counts hardlinked files

This document explains how DirSize over-reports for directories that contain hardlinks (Time Machine local snapshots, pnpm content-addressable storage, certain backup trees) and decides how much dedup to add.

On this page: Glossary · The picture · Three options · Recommendation

Glossary

inode
The actual on-disk object representing a file's data and metadata. A single inode holds the bytes; a "filename" is a directory entry that points at an inode. Different names can point at the same inode — that's a hardlink.
hardlink
A second directory entry pointing at an existing inode. The file appears in two places but takes up only one copy on disk. Deleting one name decrements the inode's link count; the bytes stay on disk until the last name is removed.
symlink (soft link)
A small file whose content is the path to another file. Different from a hardlink: a symlink can cross filesystems and survive its target being deleted (it just becomes broken). spacestation skips symlinks already, so they're not the bug.
(dev, ino)
A two-part identifier that uniquely names an inode across the whole running system: the device ID of the filesystem plus the inode number within that filesystem. Two unrelated filesystems can each have an "inode 12345", so the device number is what disambiguates.
os.FileInfo
Go's filesystem metadata struct. Its Size() method returns "the size of the file as named" — i.e. the size attributed to this name, not to the underlying inode. For a hardlinked file, every name reports the same size, even though deleting any one name frees zero bytes (the inode keeps the data alive).
CAS (content-addressable storage)
A store where files are keyed by a hash of their content. Identical content lives in exactly one place. pnpm uses this: a package's files live once in ~/Library/pnpm/store, and every project's node_modules contains hardlinks back to the store. This is also Time Machine's APFS clone model in spirit.
DirSize
The function in internal/scan/size.go that walks a directory and sums FileInfo.Size() for every regular file it sees. Today it counts every hardlinked file once per name, not once per inode.
logical size vs physical size
Logical = "sum of file sizes as you'd see them in ls." Physical = "actual bytes occupied on disk." For a tree with no hardlinks they're equal. For a CAS or snapshot tree, logical can be many times larger than physical.
syscall.Stat_t
Go's Unix-specific stat structure that exposes the device ID and inode number. os.FileInfo hides these — you have to type-assert info.Sys() to *syscall.Stat_t on Linux/macOS to read them.

The picture

Here's a concrete example. Picture a single pnpm project on disk. The project uses react, react-dom, and @types/react across a dependency tree where 14 sub-packages all pull react as a peer dependency.

~/Library/pnpm/store/v3/files/                  ← THE CAS, one copy per file
  88/                                            (inode 90001)  react/index.js     14 KB
  a2/                                            (inode 90002)  react/cjs/...      82 KB
  ...                                            ~30,000 unique inodes, ~1.8 GB total

~/projects/acme-api/node_modules/.pnpm/
  react@18.2.0/node_modules/react/
    index.js   ──hardlink──▶ inode 90001 (14 KB, ALSO named in store)
    cjs/...    ──hardlink──▶ inode 90002 (82 KB, ALSO named in store)
  react-dom@18.2.0/node_modules/react/
    index.js   ──hardlink──▶ inode 90001  (SAME inode again)
    cjs/...    ──hardlink──▶ inode 90002
  ... 12 more sub-packages, all hardlinking back to the same react inodes

When the user runs spacestation, the candidate list includes ~/Library/pnpm/store as a system path probe AND ~/projects/acme-api/node_modules as a project walk match. DirSize is called once per candidate. Here's what each call sees and reports:

CandidateWhat DirSize walksWhat it reports todayWhat it should report
~/Library/pnpm/store ~30,000 unique files, each named once inside the store 1.8 GB (correct by coincidence — store has no within-store hardlinks) 1.8 GB
node_modules of one project react's index.js appears 14 times (once per sub-package); same for every shared dep ~3.4 GB (14× inflation on the heavily-shared files) ~280 MB (unique inodes within this tree)
Why this is a real bug: The user sees node_modules listed at 3.4 GB, gets excited, deletes it — and finds out only ~280 MB was actually freed, because the 14 hardlinks to each store file just lost one name; the inode still has the store name alive. The "reclaim" headline number is overpromising.

Beyond pnpm, the same pattern shows up in:

  • Time Machine local snapshots. APFS clones are inode-shared with the live filesystem. A snapshot of ~/Library appears to contain a full copy but is mostly shared inodes.
  • rsync --link-dest backup trees. Each daily backup hardlinks unchanged files to the previous day. du reports the size as if each day were a full copy.
  • Photo libraries, certain databases. Anything that does its own snapshotting via hardlinks.

Three options

Option A — per-call inode dedup

Each DirSize call maintains its own map[uint64]struct{} keyed by (dev << 32) | ino. Before adding a file's size to the running total, check the map; skip if already seen, otherwise add. The map is local to one call, so each candidate's reported size is independent.

ProsCons
  • Each candidate's size becomes self-consistent: "if I were to follow the link count semantics, deleting this whole tree would free exactly this much."
  • Fixes the worst case (the 14× pnpm node_modules inflation).
  • Simple. ~15 lines added, no cross-call coordination, no cache complications.
  • Test is straightforward: build a tiny tree, hardlink one file twice, call DirSize, assert the size equals one copy.
  • Doesn't deduplicate across candidates. If the user selects both node_modules (now 280 MB) and ~/Library/pnpm/store (1.8 GB), the aggregate says "you'll reclaim 2.08 GB", but deleting both actually frees ~1.8 GB (the store dominates).
  • Adds a small platform-specific block: reading (dev, ino) requires type-asserting info.Sys() to *syscall.Stat_t. Fine on macOS/Linux, but cements the macOS-only nature of the binary (which Hy2 will address separately anyway).

Option B — per-scan inode dedup

One shared map[uint64]struct{} protected by a mutex, threaded through every DirSize call in a single scan.Run. First call to encounter an inode counts it; subsequent calls see it as already counted and skip. The reclaim numbers become honest about cross-candidate sharing.

ProsCons
  • The aggregate reclaim number is now truthful: "if you select all of these, you'll free X" matches reality.
  • Handles the pnpm case correctly end-to-end. The store and the project node_modules collectively report ~1.8 GB, not 5.2 GB.
  • Order-dependent sizes. Whichever candidate gets walked first "claims" the shared inodes; later candidates report a fraction of what they'd report on their own. Walks run in parallel, so order isn't even deterministic — two scans of the same disk can yield different per-candidate sizes.
  • Per-candidate sizes become misleading in isolation. The user sees node_modules: 12 MB but the moment they untick the pnpm store candidate, the real reclaim from node_modules alone would be 280 MB. The number has no useful interpretation as a standalone value.
  • Interacts badly with the persistent size cache (~/.config/spacestation/sizes.json): cached values would depend on what else was scanned in the same session. Either we don't cache (slower) or the cache becomes incoherent.
  • Forces locking on a hot path. Every regular file requires a mutex acquire on the shared map. Modest but real perf cost.

Option C — do nothing

Leave DirSize reporting "logical" sizes. Document the over-reporting in the README and the candidate detail pane. Smart probes already handle pnpm via pnpm store prune, which estimates reclaim from the tool itself; users who want accurate pnpm numbers should rely on that.

ProsCons
  • Zero code change. Zero risk.
  • For users without heavy hardlink trees (most non-pnpm, non-Time-Machine setups), the numbers are already correct.
  • The headline reclaim number stays misleading for anyone using pnpm, Time Machine local snapshots, or hardlink-based backups — exactly the audience that has the most to reclaim.
  • Trust erosion: when a user deletes 3.4 GB of node_modules and Finder only reports 280 MB freed, the tool looks broken.
  • The fix is small enough that "do nothing" is hard to defend.

Recommendation

Option A — per-call inode dedup

This is the right scope for what spacestation is. Each candidate's number becomes locally truthful: "delete just this directory and you'll free this much." That's the question a user reasonably expects each row to answer. Aggregate across candidates is still approximate, but in the common case it's the worst offender (heavily-shared node_modules) being fixed individually that closes most of the gap.

Why not Option B: the order-dependence is disqualifying. A scan result where node_modules reports "280 MB" or "12 MB" depending on which goroutine raced first is worse than the current honest-but-inflated number. The cache incoherence makes it worse still.

Why not Option C: the bug is real and dramatic on the exact audience the tool is built for. The fix is ~15 lines of mechanical code plus a synthetic-tree test. Skipping it for "documented limitation" reasons would be cowardice.

Looking for: a thumbs-up on Option A, or a steer toward B/C.
