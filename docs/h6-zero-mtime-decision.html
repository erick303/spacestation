H6 — zero-mtime candidates should not auto-select

This document walks through finding H6: a directory whose modification time we couldn't read silently gets armed for deletion, and decides how aggressively to fix it.

On this page: Glossary · The picture · Three options · Recommendation

Glossary

mtime
Modification time — a timestamp the filesystem records every time a file or directory is changed. On macOS / Linux it's a Unix epoch second + nanoseconds.
LastTouched
A function in internal/scan/recency.go. Given a directory, it returns "when was anything in here last touched" — defined as the most recent mtime among the directory's immediate children (or the directory's own mtime if it's empty).
time.Time{} (zero value)
Go's default time.Time when no value was ever set. It represents January 1 of year 1 CE — about 2025 years ago. Subtracting it from the current time gives a duration of roughly 739,000 days.
Candidate
A struct in internal/scan/types.go representing a single piece of reclaimable disk: a directory, the size in bytes, when it was last touched, what category it belongs to, and a "safety" flag (regenerable vs. user content).
SafetyRegenerable
A safety classification meaning "whatever tool produced this directory will recreate it on demand, so it's safe to delete." node_modules, .next, ~/.cargo/registry all fall here.
score.Apply
A function in internal/score/score.go that walks the list of candidates and decides which ones should be auto-selected when the TUI opens. The selection rule for Regenerable items is "stale for at least 30 days" (configurable via default_select_min_age_days).
auto-selected
When spacestation launches, the candidate list comes up with checkboxes already ticked for "obviously safe" entries — Trash always, old Downloads, old regenerable build dirs. The user can untick before hitting enter, but if they don't pay close attention they trust the defaults.
EACCES
"Permission denied" — a kernel error returned when the process can't read a file or directory due to permissions.

The picture

Here's a real scenario. The scanner has just emitted a Candidate for ~/Library/Caches/com.apple.bird (some Apple sync daemon's cache, categorised CatSystemCache, Safety Regenerable). The scanner then calls LastTouched on that path to fill in the recency field.

Walk through what happens when os.Stat fails on this one path — maybe a sandbox kicked in, maybe the cache directory was deleted between the scanner emitting it and LastTouched running, maybe it's a system path that needs Full Disk Access:

StepState
Scanner emits Candidate for ~/Library/Caches/com.apple.bird Size: 240 MB. Category: CatSystemCache. Safety: Regenerable.
Scanner calls LastTouched("~/Library/Caches/com.apple.bird") os.Stat returns EACCES. LastTouched returns time.Time{}.
score.Apply processes the candidate age = now.Sub(time.Time{}) ≈ 739,000 days.
Auto-select check fires age > 30 days → true. Candidate marked Selected: true.
User opens the TUI Row reads: [x] ~/Library/Caches/com.apple.bird 240 MB Stale 739000d, regenerable
User hits enter trusting the defaults Directory we couldn't even stat goes to the Trash.
Why this is uncomfortable: the tool's safety story is "we read mtime, we age-out anything that's been quiet for ≥30 days." When mtime is missing, we shouldn't apply that story — we should say "we don't know how old this is, ask the user." Today's behaviour does the opposite: it treats unknown as maximum-stale.

There's a cosmetic mirror: if mtime is in the future — clock skew, a restored backup with weird timestamps, an NTP glitch — then age is negative. negative > 30 days is false, so it doesn't auto-select. But the reason text shows Active (-5d) — may rebuild from scratch if deleted, which is ugly. Not dangerous, just sloppy.

Three options

Option A — minimal patch (zero-mtime only)

Add one guard at the top of the score.Apply loop. If LastTouched is the zero value, set a "unknown age — not auto-selecting" reason and continue. Leave the future-mtime cosmetic bug alone. Don't add tests.

ProsCons
  • ~3 lines of code; trivially reviewable.
  • Fixes the actual safety issue: nothing auto-selects without a real mtime.
  • No new abstractions; the existing switch still does the work it did.
  • Future-mtime entries still render as (-5d) in the reason text. Confusing for anyone with clock skew.
  • No regression guard — a future refactor of the loop could accidentally drop the zero-check and reintroduce the bug.
  • Leaves score at zero test coverage (the finding doc names this as the canary that "would have caught H6 today").

Option B — recommended scope (zero + future + test)

The guard from Option A, plus: clamp negative ages to 0 before formatting reasons, so a future-mtime entry reads Active (0d) instead of (-5d). Also add a small table-driven test for score.Apply covering: zero mtime, future mtime, stale Regenerable, recent Regenerable, Trash, Downloads. score.Apply is pure (no FS, no exec, no time mocking) so the test stays cheap.

ProsCons
  • Closes both bugs the finding called out: zero (dangerous) and future (cosmetic) in one go.
  • The test makes the rule machine-readable. Any future change to Apply that flips the zero-mtime behaviour fails CI.
  • Starts filling the test-coverage gap that Hy4 (later in the list) is dedicated to. Earlier coverage = less pressure on the Hy4 cycle.
  • Still small (~30 lines added including the test).
  • Slightly larger commit than strictly required to close the safety bug.
  • "Cosmetic mirror" fix and "test scaffolding" technically aren't the H6 finding — purists could argue they belong in their own commits.

Option C — defensive overhaul (fix LastTouched too)

Treat the bug as being upstream in LastTouched: change it to never return the zero value. Either fall back to time.Now() on stat failure (treats unknown as "just now", which means Regenerable items won't auto-select unless they have other signal), or return a sentinel and have callers explicitly check. Then also do everything in Option B.

ProsCons
  • Belt-and-suspenders: even if a future caller forgets to check, LastTouched can't poison them.
  • The semantics of LastTouched become unambiguous — "this always returns a usable timestamp."
  • Changes the contract of LastTouched. Anything else that calls it (the TUI's "89d ago" rendering, possibly future code) inherits the new semantics — some of them may want to know "we don't actually know" rather than getting a fake answer.
  • Falling back to time.Now() hides errors instead of surfacing them. The user sees "just now" in the age column for a directory we couldn't read; that's a different kind of dishonest.
  • Bigger diff, more places to think about. Easier to land in a single PR that's actually focused on this.

Recommendation

Option B — zero + future + test

The scope is right: it closes both bugs the finding actually called out, and it lays down a regression guard for the loop that mis-handled them. score.Apply is one of the cheapest things in this codebase to test (pure function, no FS, no time mocking — just craft Candidates and assert), so the cost-benefit on adding a test now versus during the Hy4 cycle later is overwhelmingly in favour of now.

Why not Option A: it leaves (-5d) in the user-facing reason text, and it leaves zero test coverage for the function whose bug we just fixed. Both are small things, but they're the difference between "patched" and "patched and guarded."

Why not Option C: changing LastTouched's contract is the wrong layer. The semantics "zero means we couldn't read it" is actually fine — the caller's job is to handle that gracefully. Replacing the zero with a fake time.Now() trades one quiet wrongness for another and complicates anything else that wants to know whether the timestamp is real.

Looking for: a thumbs-up on Option B, or a steer toward A/C.
